WCAG Conformance

Accessibility conformance documentation for the Universal Design System. Targeting WCAG 2.2 Level AA across all 9 palettes, 43 components, and both light and dark modes.

Conformance Overview

The Universal Design System targets WCAG 2.2 Level AA conformance. Accessibility is enforced at the token level, validated by automated tooling on every build, and documented per component.

Every palette in UDS is designed to meet WCAG 2.2 Level AA contrast requirements: 4.5:1 for body text, 3:1 for large text and UI components. Contrast checks run on every build via npm run audit, covering 6 color pairs per palette across both light and dark modes (9 palettes × 6 pairs × 2 modes = 108 checks). Components are built with semantic HTML, ARIA attributes, keyboard support, and visible focus indicators as baseline requirements.

WCAG 2.2 New Success Criteria (Level AA)

WCAG 2.2 introduces three new Level AA success criteria that the Universal Design System addresses. These criteria strengthen requirements for focus visibility, pointer input alternatives, and minimum target sizes.

SC 2.4.11

Focus Not Obscured (Minimum)

When a user interface component receives keyboard focus, the component is not entirely hidden due to author-created content. All interactive UDS components ensure focused elements remain at least partially visible.
  • Sticky headers do not fully obscure focused elements
  • Modal focus traps keep focus within visible boundaries
  • Dropdown and tooltip overlays do not cover the trigger element
  • Scroll-into-view behavior ensures focused content is visible
SC 2.5.7

Dragging Movements

All functionality that uses a dragging movement for operation can be achieved by a single pointer without dragging, unless dragging is essential. UDS components provide single-pointer alternatives.
  • File Upload dropzone accepts click-to-browse as alternative to drag-and-drop
  • Date Picker range selection operable via keyboard and single tap
  • No component requires drag as the only input method
SC 2.5.8

Target Size (Minimum)

The size of the target for pointer inputs is at least 24 by 24 CSS pixels, except where an equivalent control meets the minimum, the target is inline, the user agent determines the size, or the presentation is essential. All interactive UDS components meet the 24×24px minimum.
  • Buttons: smallest size (sm) is 36px height
  • Form inputs: smallest size (sm) is 36px height
  • Checkboxes and radios: label click area meets 24×24px minimum
  • Toggle switches: 48×28px minimum target area
  • Interactive component sizes documented in components.csv via min_target_size column

Testing Methodology

Conformance is validated through a combination of automated and manual testing processes.

Automated

wcag-audit.py

Custom Python validator that checks 108 contrast ratios across all 9 palettes in both light and dark modes. Runs on every build via npm run audit.
  • 6 color pairs per palette per mode
  • 4.5:1 minimum for body text
  • 3.0:1 minimum for large text and UI
  • Results written to audits/a11y-audit.json
Automated

axe-core DOM Audit

Runtime DOM analysis using the axe-core accessibility engine to detect ARIA misuse, missing labels, invalid roles, and color contrast violations in rendered HTML.
  • All docs pages scanned
  • Component sandbox validated
  • Catches runtime-only issues
Automated

APCA Analysis

Advanced Perceptual Contrast Algorithm analysis provides supplementary contrast data beyond WCAG 2.2 AA requirements. APCA scores are tracked for all palette color pairs.
  • Lc (Lightness Contrast) values computed
  • Font-size-aware contrast thresholds
  • Dark mode perceptual accuracy
Manual

Keyboard Navigation

All interactive components are tested for complete keyboard operability. Tab order, focus management, and keyboard shortcuts are validated.
  • Tab/Shift+Tab traversal
  • Enter/Space activation
  • Arrow key navigation (menus, tabs)
  • Escape to dismiss (modals, dropdowns)
Manual

Screen Reader Testing

Components tested with VoiceOver (macOS/iOS) and NVDA (Windows) to verify correct announcement of names, roles, states, and live region updates.
  • Landmark navigation
  • Heading hierarchy
  • Live region announcements
  • Form field associations
Manual

Zoom and Reflow

All pages tested at 200% and 400% zoom. Content reflows to a single column at 320px CSS width without horizontal scrolling or content loss.
  • 200% zoom baseline
  • 400% zoom verification
  • 320px reflow check
  • No information loss at any zoom level

Known Limitations

Transparency about gaps is essential for accessibility compliance. The following are known limitations that may affect conformance in specific contexts.

Medium

APCA Dark Mode Gaps

The WCAG 2.2 contrast algorithm (relative luminance) and the newer APCA algorithm can produce different results for dark-on-dark color combinations. Some dark mode palette pairs that pass WCAG 2.2 AA may score below recommended APCA thresholds, particularly for secondary and tertiary text. APCA is not yet a W3C recommendation, so WCAG 2.2 AA remains the conformance target.
Medium

Toast Auto-Dismiss Timing

Toast notifications have configurable auto-dismiss durations. The default timing may not provide sufficient reading time for users with cognitive disabilities or those using screen readers. Implementors should ensure error and warning toasts are not auto-dismissed, or provide a mechanism to pause/extend the timeout (WCAG SC 2.2.1 Timing Adjustable).
Low

Tooltip Escape Dismissal

The Tooltip component specifies appearance on hover and focus, but does not yet explicitly require dismissal via the Escape key (WCAG SC 1.4.13 Content on Hover or Focus). Implementors should add Escape key handling to dismiss tooltips without moving focus.
Low

Audit Color Value Resolution

The automated contrast audit (wcag-audit.py) currently reports all 108 checks as "skipped" with reason "Color values not resolved." This occurs because the audit script cannot resolve CSS custom property values at build time. Contrast validation is supplemented by visual inspection and browser-based testing tools. A runtime color resolver is planned.
Low

Testimonial Video Variant

The Testimonial component's video variant requires keyboard-accessible media controls. The component specification defines the ARIA patterns but does not ship a reference implementation for custom video controls. Implementors must ensure play/pause, volume, and captions are keyboard operable.
Low

Custom Select Variant

The Select component includes a "custom" variant alongside the native variant. While the native variant inherits full browser accessibility support, the custom variant requires careful ARIA implementation by consumers. The native variant is recommended as the default for maximum accessibility.
